## 4. 協作作業準則—v2.0
- **代理優先**：任務由 AI/代理先行完成開發、部署與設定；僅當所有代理路徑失敗時，才請使用者於本地區動作，且代理必須提供步驟指示。
- **SSoT**：GitHub 傳彌所重複同改及程式碼最終政凱；任何自動尾章歸固作事最終要回寫到傳彌。
- **授權分級**：
  - L2 （預設）：代理擁有 repo push 權與 Vercel 專案權，可直接部署與設定網域。
  - L1：使用者登入後代理接手。
  - L0：所有代理路徑失效時，由使用者本地動作，代理提供明確步驟。
- **DoR / DoD**：
  - DoR：確認目標、範圍、授權層級與 SSoT 後開始作業。
  - DoD：網站可通過自有網域與 SSL 正常存取；表單寄信成功；技術文件(三份) 更新並提交到 `/docs` ；保留回歸標籤。
- **互動節點協議**：代理要求使用者登入或授權時，應使用固定提示語「請在畫面完成登入，完成後回覆『已登入』」。部署完成後使用者需回覆「核准」方算驗收通過。

## 6. 部署與維護流程
1. **更新程式**：將 `index.html` 與相關檔案提交至 GitHub 主分支。
2. **自動部署**：Vercel 連接 GitHub 後自動建置並發佈；如需備換，可臺時使用 Netlify Drop 發佈。
3. **網域綁定**：在註冊商設定 A 紀錄 `76.76.21.21` 及 CNAME `cname.vercel-dns.com`；於 Vercel 專案中新增 `sibi-advisory.com` 與 `www.sibi-advisory.com`，取消 redirect 勾選，並設定 Primary 網域。
4. **驗證與測試**：通過 DNS 生效後，檢查所有 URL，驗證 SSL 憑證、表單提交及 SEO 標籤。
5. **文件更新**：完成部署後立即更新 `/docs` 中的技術規格、開發附錄與移交清單，並在 `changelog.md` 記錄版本變更。
6. **回歸策略**：每次部署創建 Git tag；如部署失敗，可通過 Vercel 回復上一版本或 git reset/tag。

## 7. 測試與驗收
- 跨設備 RWD 正確顯示，漢堡選單可用。
- 表單必填驗證正常，寄信成功時顯示成功訊息。
- Lighthouse SEO 分數 ≥ 90；包含 title、description、OG 標籤與 canonical link。
- JSON-LD 結構化資料驗證通過。
- 所有資源皆由 HTTPS 提供。

## 8. 後續擴充建議
- **CMS 組合**：可接入 Sanity、Supabase 等內容管理系統，讓非技術人員在後臟修改文案與圖片。
- **多語系支援**：擴帶英文版與其他語言，使用 i18n 檔管理文字。
- **流量分析**：加入 Google Analytics 或其他無 cookie 的結足平台，以了解用戶行為。
- **安全防護**：對 Formspree 加入 reCAPTCHA v3 或使用蜂蜜欄位防止垃圾訊息。
- **持續整合/部署 (CI/CD)**：通過 GitHub Actions 配合 Vercel CLI 進行自動部署與回歸，以及更新文件。
